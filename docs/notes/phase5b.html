Phase 5B：让 agent 会改、会跑，但先学会刹车

Phase 5B 把工具循环从只读 read_file 扩展到 write_file 和 run_shell。重点不是让模型马上拥有完整自治能力， 而是建立风险分级：安全能力直接执行，高风险 shell 只生成 pending approval 和审计记录。

1. 业务问题

coding agent 要能完成真实工程任务，必须能写文件、跑测试、读取结果。但这两类能力会改变项目状态， 尤其 shell 命令可能删除文件、修改权限、执行远程脚本。因此 Phase 5B 的产品目标是： 让工具进入统一 runtime loop，同时不能让模型静默破坏 workspace。

read read_file

只读工具，可直接执行，但仍然必须复用 workspace boundary。

mutating write_file

写入工具，采用 whole-file overwrite，避免第一版就引入模糊 patch 语义。

permission run_shell

安全命令可执行；危险命令、内部路径访问和明显路径逃逸返回 approval_required，并写入审计日志。

2. 设计边界

模块 职责 不能做什么
ChatRuntime 收集 tool_call，调用 registry，发出 tool_result，再回传 function_call_output。 不硬编码具体工具，不直接读写文件或运行 shell。
ToolRegistry 集中管理工具定义、工具元数据和本地 handler。 不绕过 service 层安全边界。
workspace_service 继续作为文件读写的唯一边界，阻止 path escape 和内部路径访问。 不因为模型调用就放宽 .git、.kodeks 等限制。
audit_service 为危险 shell 生成 approval_id 并写入 .kodeks/tool_audit.jsonl。 不保存完整文件内容，不提供批准执行能力。

3. 执行流

User request
  -> ChatRuntime sends tool definitions: read_file, write_file, run_shell
  -> OpenAI Responses stream emits tool_call
  -> ChatRuntime calls ToolRegistry.execute(...)
  -> read_file/write_file reuse workspace_service
  -> run_shell checks dangerous command and path policy
  -> safe shell executes; risky shell returns approval_required
  -> ChatRuntime emits tool_result
  -> ChatRuntime sends function_call_output back to provider
  -> provider streams final answer

这里的关键取舍是：Phase 5B 先把 permission gate 放进工具执行路径，而不是等 Phase 6 才意识到 shell 已经能被模型随意调用。

4. 验证

验证项 覆盖点
test_default_registry_exposes_phase5b_tools registry 同时暴露三个工具定义。
test_execute_write_file_uses_whole_file_overwrite 写入语义明确，能报告覆盖和字节数。
test_execute_write_file_reuses_workspace_boundary 模型不能写入 .kodeks 等内部路径。
test_execute_run_shell_dangerous_command_requests_approval_and_audits 危险命令、内部路径和路径逃逸不执行，产生 pending approval 和 audit log。
test_runtime_pauses_dangerous_shell_tool_and_records_audit runtime 层能把 approval_required 作为 tool_result 回传给模型。

5. 面试讲法

q1: Phase 5B 解决了什么工程问题？
a1: 它把 coding agent 从只会读文件，推进到能写文件、跑命令；同时建立了工具风险分级， 避免模型直接拥有无约束 shell。

q2: 为什么 write_file 先用 whole-file overwrite？
a2: 因为第一版需要语义清楚、容易测试、失败边界明确。diff-based edit 更适合在 approval 和 patch 展示能力成熟后再做。

q3: 为什么危险 shell 不直接执行？
a3: shell 是 coding agent 里最高风险的工具。Phase 5B 只生成 approval_id 和审计记录，Phase 6 再实现批准/拒绝接口，这样系统不会在权限产品化前静默破坏项目。

6. Phase 6 Handoff

下一阶段应该围绕 approval flow 产品化：新增批准/拒绝接口、根据 approval_id 恢复原命令、记录最终 decision，并验证危险命令没有审批永远不能执行。
