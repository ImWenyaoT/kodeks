Phase 6：把危险动作交还给人类决策

Phase 5B 只会把危险 shell 暂停成 approval_required。 Phase 6 把这个 pending 状态产品化：人类可以查看、批准或拒绝；批准后命令只执行一次； 每一步都追加审计记录。

1. 业务问题

一个真实 coding agent 不能因为模型想运行 rm -rf 或访问内部路径就直接照做。 但它也不能永远卡在 pending 状态。Phase 6 的目标是把“暂停”升级成“可决策、可恢复、可追踪”的 approval flow。

pending Ask

Phase 5B 生成 approval_id，记录原始命令和原因。

approved Approve

批准后通过 dedicated approved-command path 执行一次，并记录结果。

rejected Reject

拒绝后不执行命令，记录拒绝原因，approval 不能再次消费。

2. 状态机

pending
  -> approve -> approved -> executed | execution_timeout
  -> reject  -> rejected

resolved approvals cannot be processed again

这里刻意不做 always allow、持久规则和 UI prompt。第一版只保证： approval 可以被查询，只能被消费一次，并且每次状态变化都写入 JSONL。

3. API

Endpoint 作用 错误边界
GET /api/approvals/{approval_id} 读取 approval 最新状态。 不存在返回 404。
POST /api/approvals/{approval_id}/approve 批准并执行 pending shell command。 不存在 404；已处理 409；超时 408。
POST /api/approvals/{approval_id}/reject 拒绝 pending shell command。 不存在 404；已处理 409。

5. 面试讲法

q1: Phase 6 解决了什么工程问题？
a1: 它把模型请求危险命令这件事，从“模型直接执行”改成“模型申请、人类决策、系统审计、一次性恢复执行”。

q2: 为什么 approval 只能消费一次？
a2: 因为批准是对某个具体风险动作的一次性授权。重复执行会让权限语义失真，也会造成不可预期的副作用。

q3: 为什么还不做 always allow？
a3: 持久规则会把一次判断升级成长期策略，风险更高。先做 one-shot approval，能把状态机、审计和恢复执行测扎实。
