Phase 4 Goal Architecture Rules Verification Interview Phase 5
kodeks · conversation state

让 coding agent 从单轮输出进入连续任务

Phase 4 把 OpenAI 的 previous_response_id 包成 kodeks 自己的 session runtime。 用户可以用同一个 session_id 连续交代任务，后端会自动续上上一轮响应。

1. 真实业务问题

用户场景

用户不会只问一次。他会让 agent 先读代码、再解释、再修改、再验证。

没有状态的问题

每轮都变成一次性问答，模型不知道上一轮做了什么，多步任务会断裂。

本阶段解法

用 SQLite store 维护 session_id -> previous_response_id，同时把存储细节隔离在接口后面。

2. 架构边界

api/routes/chat.py
  复用进程级 ChatRuntime，负责 SSE 传输

runtime/chat_runtime.py
  解析 session_id，选择 previous_response_id，更新 session store

runtime/session_state.py
  SessionStateStore 协议，SQLite 默认持久化，InMemory 用于测试

services/api/openai_responses.py
  outbound API client，只负责调用 OpenAI Responses API

这里的关键不是 SQLite 本身，而是把“会话策略”放在 runtime 层。 route 不知道 OpenAI 和 SQL 细节，OpenAI client 也不知道业务 session 规则。

3. 状态规则

规则 原因
显式 previous_response_id 优先。 客户端显式指定上下文时，不能被服务端缓存覆盖。
没传 previous_response_id 时，用 session_id 查 store。 这让同一个会话能自然连续。
没传 session_id 时，runtime 生成 session_created 事件。 客户端能拿到新会话 id，并在下一轮继续使用。
只有 response_completed 才更新 store。 失败请求不能污染有效上下文。

4. 验证记录

验证项 结果 意义
store 单测 通过 覆盖 get/set/clear/覆盖更新，以及 SQLite 跨实例持久化。
runtime 单测 通过 覆盖显式 previous 优先、store fallback、completed 更新、error 不更新、自动创建 session。
route 单测 通过 同一个 session_id 第二轮会带上第一轮 response_id。
真实 OpenAI key 未消耗 本阶段先锁定 runtime 行为；真实 provider 仍需兼容 Responses streaming。

5. 面试问答

q1: Phase 4 解决了什么真实业务问题？

a1: 它让 coding agent 支持连续多轮任务。用户不需要每轮重复上下文， 服务端会把上一轮 response_id 作为下一轮上下文。

q2: 为什么 Phase 4 可以用 SQLite？

a2: 因为它解决的是很真实的产品问题：服务重启后 session 仍可恢复。 但 runtime 只依赖 SessionStateStore 协议，所以未来换成 Conversations API、Redis 或 Postgres 也不需要改业务编排。

q3: 这和 memory/RAG 有什么区别？

a3: conversation state 解决同一任务链路里的上下文延续；memory 是跨任务长期偏好； RAG 是外部知识注入。Phase 4 只解决第一类问题。

q4: 这如何体现“驾驭 AI、提升产出”？

a4: 我不是只调用模型 API，而是把 API 的 state 能力产品化成自己的 session runtime， 并用测试锁定失败边界，让 AI 能稳定参与多步工程任务。
