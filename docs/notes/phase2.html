Shell Harness：让 agent 能验证，但不能失控

Phase 1 让 agent 能读写文件。Phase 2 让 agent 能运行命令，拿到 stdout、stderr 和 exit code，从而形成“修改代码 -> 运行验证 -> 继续修”的闭环。 但 shell 是高风险能力，所以第一版必须包含 workspace cwd、timeout 和危险命令拦截。

1. 业务需求

对 coding agent 来说，能改代码只是半成品。真实开发闭环里，它还要能运行命令检查结果： 看目录、跑编译、跑测试、读取错误输出，然后根据反馈继续修改。

需求 为什么重要 本阶段实现
运行项目命令 让 agent 能从环境反馈中学习，不只是静态改文件。 POST /api/shell/run
结构化返回 agent loop 后续要判断成功、失败、stderr 错误文本。 command、exit_code、stdout、stderr
防止卡死 命令可能长时间运行，服务不能被单个请求挂住。 timeout 后返回 HTTP 408
危险命令拦截 删除、重置、提权、管道安装等操作必须先要用户确认。 approval_required=true

2. 架构设计

api/routes/shell.py
  route 层：HTTP 请求、timeout 状态码、响应结构

services/shell_service.py
  service 层：危险命令识别、subprocess 执行、stdout/stderr 捕获

core/config.py
  config 层：workspace root，命令默认在这里执行

这个分层让 shell 能力和 HTTP 表达解耦。后续进入 OpenAI tool calling 时， run_command 可以复用为 agent tool 的底层能力，但不会在 Phase 5A 直接暴露给模型。 shell 工具要等 approval id、审计日志和更强 command policy 完成后再接入。

3. 安全策略

受控执行环境

所有命令默认在 workspace root 下执行。这样 pwd、 ls、未来的测试命令都围绕当前项目，而不是用户机器的任意目录。

超时保护

subprocess.run 设置 timeout。长时间命令会被中断，并由 route 映射成 408， 让调用方知道这是“卡死”，不是普通执行失败。

当前版本的危险命令识别是第一版 guardrail，不是最终 sandbox。它适合 MVP 和面试展示， 后续还要演进成 approval id、用户确认、命令审计日志，以及更严格的 shell parsing。

5. 面试问答

q1: Phase 2 解决了什么真实工程问题？

a1: 它让 coding agent 从“会改文件”升级到“能验证修改”。agent 可以运行命令， 拿到 stdout/stderr/exit code，再根据反馈继续修。

q2: 为什么 shell 不能直接暴露给模型？

a2: shell 权限太大，可能删除文件、重置 Git、提权、执行远程脚本或卡死服务。 所以我用 shell harness 统一限制 cwd、timeout，并对危险命令返回 approval_required。

q3: timeout 为什么重要？

a3: agent 可能运行测试、服务启动命令或错误命令。如果没有 timeout，一个请求就可能让后端挂住。 我把 timeout 映射成 408，让 agent loop 能区分“命令失败”和“命令卡死”。

q4: 你的危险命令识别有什么局限？

a4: 现在是 regex-based guardrail，适合 MVP，但不是完整安全沙箱。后续要加入更严格的命令解析、 approval id、审计日志，以及对 shell metacharacters 的系统处理。

6. Phase 3 Handoff：Streaming-First Event Baseline

Phase 1 和 Phase 2 已经把 agent 的“身体”做出来：能读写文件、能运行命令、能阻止明显危险操作。 Phase 3 接入 OpenAI Responses API 时不再先做普通 blocking chat，而是直接定义 streaming event contract。 这是因为 coding agent 后续要持续报告文本、工具调用、工具结果和错误状态。

下一步 业务价值 验收
封装 outbound API client 把 OpenAI Responses API 细节隔离在 services/api 层。 缺 key、空 stream、失败 stream 都能映射成 kodeks event。
SSE chat stream 让前端或 CLI 可以边生成边渲染。 POST /api/chat/stream 返回 text_delta 和终态事件。
runtime event contract 为 Phase 4 session 和 Phase 5 tool loop 打基础。 事件类型包含 text_delta、response_completed、error，并预留工具事件。
