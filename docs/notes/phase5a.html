kodeks · tool orchestration

把模型输出变成真实工具动作

Phase 5A 的目标不是“多接几个工具”，而是完成 coding agent 的第一个完整 tool loop： 模型请求读取文件，runtime 执行本地只读工具，再把结果交回模型继续回答。

1. 真实业务问题

用户需求

用户希望 agent 能基于项目真实文件回答，而不是凭空猜测代码内容。

工程挑战

模型不能直接碰本机文件系统。它只能提出工具调用，由 runtime 按安全策略执行。

本课产物

一个只支持 read_file 的最小闭环，为后续写文件、跑测试、approval 打基础。

2. 为什么只做 read_file

能力 Phase 5A 决策 原因
read_file include 只读、可复用现有 workspace boundary，最适合验证 tool loop。
write_file defer 会修改项目，需要先设计 overwrite、diff 和用户可理解的变更展示。
run_shell defer 命令执行风险更高，需要 approval id、audit log 和更强 command policy。

3. Runtime 流程

user input
  -> ChatRuntime builds ChatProviderRequest(tools=[read_file])
  -> OpenAIResponsesProvider streams text_delta or tool_call
  -> ChatRuntime sees tool_call(name="read_file")
  -> ToolRegistry executes workspace_service.read_file(path)
  -> ChatRuntime emits tool_result SSE event
  -> ChatRuntime sends function_call_output back to provider
  -> provider streams final text_delta and response_completed

这里最重要的分工是：provider 只翻译 OpenAI event，tool registry 只执行本地工具， runtime 负责把它们编排成一个连续工作流。

5. 课程步骤

步骤 要做什么 验收信号
1 新增最小 ToolRegistry，只注册 read_file。 能从 registry 查到工具定义和执行函数。
2 让 ChatRuntime 给 provider request 带上 read_file tool definition。 provider 单测能看到 strict function schema。
3 runtime 捕获 tool_call，执行本地读取。 workspace blocklist 仍然生效，不能读 .kodeks。
4 runtime 发出 tool_result，再提交 function_call_output。 SSE 中能看到 tool_result，随后继续收到最终文本。

7. 核心能力在教案里的位置

这些能力不是 Phase 5A 才加入教案。kodeks 的长期目标从一开始就是带 memory、multi-session、 plan mode、subagent 的 coding agent；下面的 phase 只说明实现顺序。

组件 课程位置 当前边界
protocol Phase 3 已建立事件协议，Phase 5A 继续使用 tool_call / tool_result。 协议是 agent 的共同语言，不绑定 HTTP route 或某个 UI。
multi-session/context Phase 4 已实现 session_id -> previous_response_id。 解决同一任务链路里的连续上下文，不等同于长期记忆。
tools Phase 5A 从 read_file 单工具闭环开始。 先证明工具编排，再扩展写文件、shell 和 approval。
memory Phase 4 已讲清和 context 的区别，Phase 7 进入长期 memory 设计。 长期 memory 必须可解释、可审计，不能让模型静默污染永久状态。
planning/plan mode Phase 8 进入最小 plan mode。 规划负责拆解步骤和失败后 re-plan，不直接替代工具执行。
subagent Phase 9 进入子代理编排。 子代理用于独立探索、并行分析或验证，再把结果汇总回主计划。

8. 面试讲法

q1: 你做的 tool loop 本质是什么？

a1: 我把模型的 function call 请求转成 runtime 可控的本地工具执行，并把执行结果作为 function_call_output 回传给模型，让 agent 能基于真实项目文件继续推理。

q2: 为什么不一开始就接 write_file 和 shell？

a2: 因为只读、写入、命令执行的风险等级不同。先用 read_file 打穿编排闭环， 再给高风险工具补 approval、审计和明确的写入语义，这是更稳的工程顺序。Phase 5B 已按这个 handoff 接入 write_file 与 run_shell，并把危险 shell 暂停为 pending approval。

q3: 这个阶段展示了什么工程能力？

a3: 展示的是把 LLM API 产品化的能力：不是只调用模型，而是设计 event contract、 tool registry、安全边界和可验证的 agent runtime。
