kodeks · Phase 1

Workspace 文件能力：让 agent 有手，但只能伸到该伸的地方

这一阶段不是普通 CRUD，而是在构建 coding agent runtime 的第一层能力： agent 可以看到、读取、写入项目文件，但必须被限制在 workspace 内，并且不能碰 .git、.venv、缓存目录等内部路径。

1. 业务需求

用户把一个项目目录交给 coding agent，是希望它能像初级工程师一样阅读项目、理解上下文、修改文件。 但从产品和安全角度看，agent 不能拥有整台机器的自由文件权限。

需求 为什么重要 本阶段实现
列出项目文件 让 agent 建立项目地图，知道有哪些入口和配置。 GET /api/workspace/files
读取文件 让 agent 能理解代码、README、配置和任务文件。 GET /api/workspace/read
写入文件 让 agent 能生成代码、修 bug、更新文档。 POST /api/workspace/write
限制权限 避免模型误读敏感文件或写坏内部目录。 workspace containment + internal path blocking

2. 架构设计

这一阶段刻意没有接 OpenAI。原因是：先把本地能力做成可独立验证的 service/API， 后续再把同一套 service 包装成 Responses API 的 function tools。

HTTP Client 前端、curl、TestClient 或未来的 agent 控制层。
workspace route 处理入参、状态码和错误映射。
workspace service 承载 list/read/write 和安全策略。
path policy resolve、containment、blocked internal paths。
filesystem 真实项目文件，但只暴露被允许的部分。 
api/routes/workspace.py
  route 层：HTTP 入参、状态码、错误映射

services/workspace_service.py
  service 层：文件系统操作、安全策略、可复用业务能力

core/config.py
  config 层：workspace root

3. 安全边界

第一层：防 workspace 外逃逸

用户传入路径后，不直接字符串拼接读取，而是先和 workspace root 组合， 再 resolve() 成真实绝对路径。

如果 resolved target 不在 workspace root 内，返回 403。

第二层：防 workspace 内部敏感路径

即使路径仍在项目内，也禁止访问 .git、.venv、 __pycache__、.pytest_cache、.mypy_cache、 .DS_Store。

工程品味重点：list_files、read_file、write_file 必须共用同一套路径策略。否则会出现列表里隐藏了 .git， 但读文件接口仍能读 .git/config 的安全不一致。

5. 面试问答

q1: 你这个阶段解决了什么真实工程问题？

a1: 我解决的是 coding agent 的 workspace 文件访问问题。agent 需要读取和修改项目文件， 但不能无限制访问整台机器，所以我先做了 list/read/write 三个受控文件能力。

q2: 为什么不一开始就接 OpenAI API？

a2: OpenAI API 负责模型推理和 tool call 请求，但本地文件系统能力必须由我的 runtime 负责。 如果 runtime 没有独立验证过的安全边界，接 LLM 之后风险会更难定位。

q3: 你怎么防路径逃逸？

a3: 我把用户传入的相对路径和 workspace root 拼接后做 resolve()， 然后检查 resolved target 是否仍然位于 workspace root 内。像 ../../.ssh/id_rsa 这种路径会被拒绝。

q4: 只防路径逃逸够不够？

a4: 不够。workspace 内也有不应该给 agent 访问的路径，比如 .git 和 .venv。所以我额外做了 internal path blocking。

q5: 如果面试官追问“为什么 list/read/write 共用策略”？

a5: 因为这是同一个权限模型的三个入口。如果 list 隐藏了内部文件，但 read/write 可以访问， 那就是权限不一致。统一策略能保证“看不到、读不到、写不到”的边界一致。

q6: 这个设计后面怎么接 Responses API？

a6: 后面会把 read_file、write_file 作为 function tools 注册给模型。 模型只产生工具调用意图，真正执行仍然走这套 workspace service。

6. Phase 2 Handoff：Shell Harness

Phase 1 让 agent 能改项目。Phase 2 要让 agent 能验证项目，例如运行 ls、python -m compileall、未来的 pytest。 这就是 shell harness。

下一步能力 业务价值 风险 第一版策略
运行低风险命令 让 agent 能检查文件、运行基础验证。 命令卡死、输出过大。 加 timeout，返回 stdout/stderr/exit code。
危险命令识别 避免 agent 直接删除、重置、安装或外发数据。 rm -rf、sudo、curl | sh。 先返回 approval_required，不执行。
approval flow 用户仍然可以批准高风险但合理的操作。 误批准、上下文丢失。 生成 approval id，记录原命令。
