kodeks · streaming-first baseline

把模型输出变成 coding agent runtime 事件

Phase 3 的目标不是做一个普通聊天接口，而是定义一个后续 session、memory、tool loop、 approval 和 subagent 都能复用的 streaming event baseline。

1. 业务目标

真实需求

coding agent 不能等所有输出生成完才响应。它需要边生成、边展示、边报告工具状态， 这样用户才能理解 agent 正在做什么。

本阶段边界

只接文本 streaming，不接工具调用。工具事件先在 contract 里预留，Phase 5 再真正执行。

为什么先做事件

事件协议是后续 CLI、TUI、Web UI 的共同语言。先稳定协议，比先堆 UI 更重要。

2. 架构边界

api/routes/chat.py
  HTTP/SSE adapter，只负责请求和响应

runtime/events.py
  kodeks 自己的 agent event contract

runtime/chat_runtime.py
  业务编排层，Phase 4 会在这里加入 session state

services/api/openai_responses.py
  outbound API client，把 OpenAI SDK event 翻译成 kodeks event

这次重排的核心是高内聚低耦合：route 不知道 OpenAI，services/api 不知道 HTTP 入站细节，runtime 负责把它们组合成一个 coding-agent turn。 这个命名也借鉴了 Claude Code 的 services/api 形态：它表示 outbound API client， 不是 FastAPI route。

3. Event Contract

事件 含义 后续用途
text_delta 模型正在输出文本片段。 CLI/TUI/Web 可以逐字或逐块渲染。
response_completed 一次模型 turn 正常完成，并带回 response_id。 Phase 4 用它更新 session_id -> previous_response_id。
error 缺 key、provider 失败、空 stream、无终态 stream 等异常。 调用方能清楚知道 turn 没有稳定完成。
tool_call / tool_result 暂时预留，还没有执行。 Phase 5 接 workspace/shell tools 时使用。

4. 验证记录

验证项 结果 意义
compileall 通过 新模块拆分后没有语法/import 错误。
unittest discover 通过 6 个 fake stream 测试覆盖正常、缺 key、空 stream、incomplete 和无终态。
curl /api/chat/stream 通过 FastAPI route 能输出 SSE error event，证明 HTTP adapter 仍可用。
真实 OpenAI key 未消耗 本阶段用 fake outbound API client 锁定 runtime 行为；真实 API client 需确认 base URL 支持 Responses streaming。

5. 面试问答

q1: Phase 3 解决了什么工程问题？

a1: 它把模型 streaming 输出统一成项目内部事件协议，让后续 UI、CLI、session、tool loop 都能围绕同一套事件工作。

q2: 为什么不用普通 blocking chat API 起步？

a2: coding agent 的核心体验是持续反馈：文本、工具调用、工具结果、approval pause 都是事件。 如果先做 blocking chat，后面还要推翻一次接口。

q3: 这次结构重排体现了什么设计能力？

a3: 我把 HTTP transport、agent runtime、OpenAI outbound API client 三层分开。这样未来换 CLI、 换模型 API、加 session state，都只改对应层。

q4: Phase 3 和 RAG 有什么关系？

a4: 没有直接关系。Phase 3 是 runtime event 基础设施；RAG 是外部知识注入策略。 这个项目先做 coding-agent loop，检索以后可以作为 tool 或 context provider 插进来。

6. Phase 4 Handoff：Conversation State

下一阶段不要急着做长期 memory。先做最小 session_id -> previous_response_id 状态存储，让同一个 session 的第二轮请求能自动续上第一轮 response。

下一步 业务价值 验收
内存 session store 让服务端能记住每个 session 的最后一个 OpenAI response。 完成事件后写入 store，错误事件不写入。
previous id 选择规则 支持客户端显式传入，也支持服务端自动续接。 显式 previous_response_id 优先于 session store。
连续对话测试 证明 kodeks 已经从单 turn stream 进入 multi-session runtime。 同一个 session_id 第二轮会带上第一轮 response_id。
